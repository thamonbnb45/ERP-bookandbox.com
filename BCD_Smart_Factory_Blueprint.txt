## 3. หลักการออกแบบ (Design Principles)

  1. ลดภาระสมอง — ระบบคิดแทนคน ไม่ต้องจำ ไม่ต้องคำนวณเอง
  2. แจ้งเตือนเชิงรุก — ไม่รอให้คนมาดู ระบบแจ้งผ่าน LINE เมื่อมีความเสี่ยง
  3. วัดผลทุกมิติ — คน (Skill) + เครื่อง (OEE) + งาน (Quality) + เงิน (Cost)
  4. Replicable — ออกแบบให้ปรับ config แล้วใช้กับธุรกิจอื่นได้

## 5. ข้อมูลที่ต้องเก็บ (Data Requirements)

### 5.1 ข้อมูลคน (People)
  - รายชื่อพนักงาน + แผนก + ตำแหน่ง + เงินเดือน
  - Skill Matrix (ทักษะ x ระดับ 0-3)
  - เวลาทำงาน (Clock-in/out)
  - ผลงานรายบุคคล (งานดี/งานเสีย/ความเร็ว)

### 5.2 ข้อมูลเครื่องจักร (Machine)
  - รายชื่อเครื่อง + ความเร็วมาตรฐาน (แผ่น/ชม.)
  - เวลาตั้งเครื่อง (Setup Time)
  - ประวัติซ่อมบำรุง (PM Log)
  - OEE = Availability × Performance × Quality

### 5.3 ข้อมูลงาน (Job/Production)
  - Job Order (ลูกค้า + สินค้า + จำนวน + ราคา)
  - สถานะผลิต (Kanban: รอ → ทำ → เสร็จ)
  - Production Log (เวลาจริง + ยอดดี + ยอดเสีย)
  - กำหนดส่ง vs วันส่งจริง (OTD)

### 5.4 ข้อมูลต้นทุน (Cost)
  - ค่าแรง/ชม. ต่อคน
  - ค่าเดินเครื่อง/ชม.
  - ค่าวัตถุดิบต่อ Job
  - ต้นทุนจัดส่ง

## 6. สูตรคำนวณหลัก (Key Formulas)

  OEE = Availability × Performance × Quality

  Availability = (Planned Time - Downtime) / Planned Time
  Performance  = Actual Output / (Standard Speed × Run Time)
  Quality      = Good Units / Total Units

  FTE Required   = Total Work Hours / 176 (ชม./เดือน)
  Utilization %  = Actual Work Hours / Available Hours × 100
  Backlog Days   = Total Queued Sheets / (Machine Speed × 8 ชม./วัน)
  Cost per Job   = (ค่าแรง/ชม. × ชม.) + (ค่าเครื่อง/ชม. × ชม.) + วัตถุดิบ

## 7. ขั้นตอนนำไปใช้กับธุรกิจอื่น (Replication Guide)

  ขั้นที่ 1: Clone Repository → ปรับ strategy_data.json ตามแผนกและโครงการ
  ขั้นที่ 2: ตั้งค่า Database → สร้างตารางตาม Schema ในไฟล์ migration
  ขั้นที่ 3: ปรับ Machine Config → แก้ MACHINES array ตามเครื่องจักรจริง
  ขั้นที่ 4: ปรับ Department Config → แก้ DEPARTMENTS array ตามแผนกจริง
  ขั้นที่ 5: เชื่อม LINE OA → ตั้งค่า Webhook + Channel Token
  ขั้นที่ 6: Deploy → Railway / Vercel + Neon PostgreSQL
